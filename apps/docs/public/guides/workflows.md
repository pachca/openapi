
# Сценарии

Пошаговые инструкции для типичных задач с Пачкой. Каждый сценарий описывает, какие API-методы вызывать и в каком порядке, с учётом неочевидных ограничений.

Сценарии используются как:
- **Справочник для разработчиков** — быстрый ответ на вопрос «как сделать X через API»
- **Основа для AI-агентов** — входят в [Agent Skills](/guides/ai-agents#agent-skills) для подключения к 40+ AI-агентам
- **Встроенная подсказка в CLI** — доступны через команду `pachca guide` в [CLI](/guides/cli#stsenarii)

### pachca-profile

**Получить свой профиль**

1. Получи информацию о текущем пользователе

> Возвращает `id`, `first_name`, `last_name`, `nickname`, `email`, `phone_number`, `department`, `title`, `role`, `suspended`, `invite_status`, `list_tags`, `custom_properties`, `user_status`, `bot`, `sso`, `created_at`, `last_activity_at`, `time_zone`, `image_url`.

**Проверить свой токен**

1. Получи информацию о токене: скоупы, дату создания, срок жизни

> Полезно для диагностики: какие скоупы доступны токену, когда он истекает.

**Установить статус**

1. Установи статус

**Сбросить статус**

1. Удали статус

**Получить кастомные поля профиля**

1. Получи список дополнительных полей для сотрудников
2. Получи профиль — в `custom_properties` содержатся значения полей

> Кастомные поля настраиваются администратором пространства.

### pachca-users

**Получить сотрудника по ID**

1. Получи информацию о сотруднике

> Возвращает все поля, включая `custom_properties`, `user_status`, `list_tags`.

**Массовое создание сотрудников с тегами**

1. Создай тег (если нужен)
2. Для каждого сотрудника: создай аккаунт с тегами
3. Или обнови существующего

> Создание доступно только администраторам и владельцам (не ботам). Нет отдельного эндпоинта "добавить юзера в тег".

**Найти сотрудника по имени или email**

1. Поиск по имени/email (частичное совпадение)

> Пагинация cursor-based: `limit` и `cursor` из `meta`. Для точного email — перебери страницы.

**Онбординг нового сотрудника**

1. Создай аккаунт
2. Добавь в нужные каналы
3. Отправь welcome-сообщение

> Шаг 1 требует токена администратора/владельца. Шаги 2-3 можно делать ботом.

**Offboarding сотрудника**

1. Заблокировать доступ
2. Опционально: удалить аккаунт полностью

> Приостановка (`suspended`) сохраняет данные, удаление — необратимо.

**Получить всех сотрудников тега/департамента**

1. Найди тег по названию, возьми `id`
2. Получи всех участников тега

**Управление статусом сотрудника**

1. Получить текущий статус
2. Установить статус
3. Удалить статус

### pachca-chats

**Создать канал и пригласить участников**

1. Создай канал с участниками
2. Или добавь участников позже

> `channel` — boolean, не строка. `member_ids` и `group_tag_ids` — опциональны при создании.

**Архивация и управление чатом**

1. Архивировать чат
2. Разархивировать чат
3. Изменить роль участника
4. Удалить участника
5. Покинуть чат

**Переименовать или обновить чат**

1. Обнови чат

> Для изменения состава участников используй POST/DELETE /chats/{id}/members.

**Создать проектную беседу из шаблона**

1. Создай беседу с участниками из тега
2. Отправь приветственное сообщение

> `group_tag_ids` при создании добавляет всех участников тега сразу.

**Экспорт истории чата**

1. Запроси экспорт
2. Дождись вебхука: придёт JSON с `"type": "export"`, `"event": "ready"` и `export_id`
3. Скачай файл экспорта

> `webhook_url` обязателен — POST не возвращает id в ответе. Экспорт доступен только Владельцу на тарифе «Корпорация». Макс период: 45 дней (366 при указании конкретных чатов).

**Найти активные чаты за период**

1. Получи чаты с активностью после указанной даты

**Найти и заархивировать неактивные чаты**

1. Получи чаты без активности с нужной даты
2. Для каждого чата: архивируй

### pachca-messages

**Найти чат по имени и отправить сообщение**

1. Найди чат по названию через поиск
2. Отправь сообщение в найденный чат

> `entity_type` по умолчанию `"discussion"`, можно не указывать.

**Отправить сообщение в канал или беседу (если chat_id известен)**

1. Отправь сообщение в чат

> `"entity_type": "discussion"` используется по умолчанию, можно не указывать

**Отправить личное сообщение пользователю**

1. Определи `user_id` получателя
2. Отправь личное сообщение

> Создавать чат не требуется — он создаётся автоматически

**Ответить в тред (комментарий к сообщению)**

1. Получи или создай тред, возьми `thread.id` из ответа
2. Отправь сообщение в тред

> `skip_invite_mentions: true` — не добавлять упомянутых пользователей в тред автоматически.

**Ответить пользователю, который написал боту**

1. Вебхук содержит `entity_type` — он однозначно определяет контекст: `"user"` — личное сообщение боту, `"thread"` — сообщение в треде, `"discussion"` — сообщение в канале или беседе
2. DM (`entity_type: "user"`): ответь личным сообщением
3. Тред (`entity_type: "thread"`): ответь в тот же тред
4. Беседа/канал (`entity_type: "discussion"`): inline-ответ или тред

> Если бота вызвали в треде — основной контекст в родительском сообщении треда. В вебхуке есть `thread.message_id` — получи родительское сообщение: `pachca messages get <message_id>`.

**Отправить сообщение с файлами**

1. Загрузи каждый файл — команда вернёт `key`
2. Отправь сообщение со всеми файлами

> `pachca upload` автоматически получает подпись (POST /uploads), подставляет имя файла в `key` и загружает на S3. Возвращает готовый `key` для использования в `files`.

**Отправить сообщение с кнопками**

1. Сформируй массив `buttons` — массив строк, каждая строка — массив кнопок
2. Отправь сообщение с кнопками

> `buttons` — массив массивов (строки × кнопки). Максимум 100 кнопок, до 8 в строке. Кнопка с `url` открывает ссылку, с `data` — отправляет событие на вебхук.

**Получить историю сообщений чата**

1. Получи сообщения чата с пагинацией

> Для сообщений треда используй `chat_id` треда (`thread.chat_id`). Пагинация cursor-based, не page-based.

**Получить вложения из сообщения**

1. Получи сообщение — в `files[]` каждый объект содержит `url`, `name`, `file_type`, `size`
2. Скачай нужные файлы по `files[].url`

> Вебхук о новом сообщении НЕ содержит вложений — поле `files` отсутствует. Всегда проверяй вложения через GET /messages/{id}.

**Закрепить/открепить сообщение**

1. Закрепить сообщение
2. Открепить сообщение

> В чате может быть несколько закреплённых сообщений.

**Подписаться на тред сообщения**

1. Получи или создай тред, возьми `chat_id` из ответа
2. Добавь бота в участники чата треда
3. Теперь бот будет получать вебхук-события о новых сообщениях в этом треде

> POST /messages/{id}/thread идемпотентен — безопасно вызывать повторно.

**Упомянуть пользователя по имени**

1. Определи поисковый запрос — используй фамилию (она уникальнее). Имена не склоняются в API, приводи к именительному падежу
2. Ищи среди участников целевого чата
3. Если пишешь в тред: также проверь участников родительского чата
4. Не нашёл — ищи по всей компании
5. Один результат → используй `nickname`. Несколько → уточни у пользователя
6. Вставь `@nickname` в текст сообщения

> Поиск среди участников чата точнее — пользователь явно связан с контекстом.

**Отредактировать сообщение**

1. Обнови сообщение

> Редактировать можно только свои сообщения (или от имени бота).

**Изменить вложения сообщения**

1. Получи текущие вложения из `files[]`
2. Если нужно добавить новый файл — загрузи его
3. Обнови сообщение с новым массивом `files`

> `files: []` удаляет все вложения. Если поле `files` не передавать — вложения не меняются.

**Удалить сообщение**

1. Удали сообщение

**Добавить реакцию на сообщение**

1. Добавь реакцию
2. Убрать реакцию

> `code` — emoji-символ, не его текстовое название.

**Проверить, кто прочитал сообщение**

1. Получи массив `user_id` прочитавших
2. При необходимости сопоставь с именами сотрудников

**Разослать уведомление нескольким пользователям**

1. Определи список `user_id` получателей
2. Для каждого: отправь личное сообщение

> Соблюдай rate limit: ~4 req/sec для сообщений. Добавляй паузы при большом списке.

### pachca-bots

**Настроить бота с исходящим вебхуком**

1. Создай бота в интерфейсе Пачки: Автоматизации → Интеграции → Webhook
2. Получи `access_token` бота во вкладке «API» настроек бота
3. Укажи Webhook URL для получения событий

> Бот создаётся через UI, не через API. Единственный эндпоинт для ботов — PUT /bots/{id} (обновление webhook URL). API используется для отправки сообщений от имени бота.

**Обновить Webhook URL бота**

1. Обнови webhook URL бота

> Обновлять настройки может только тот, кому разрешено редактирование бота.

**Обработать входящий вебхук-событие**

1. Получи POST-запрос на свой Webhook URL
2. Проверь подпись (Signing secret) для безопасности
3. Проверь `webhook_timestamp` — должен быть в пределах 1 минуты
4. Разбери JSON: тип события, данные
5. Для полной информации запроси сообщение — особенно для вложений (`files[]`)

> Вебхук содержит минимум данных — файлы (`files`) в нём отсутствуют.

**Разворачивание ссылок (unfurling)**

1. Создай специального Unfurl-бота и укажи отслеживаемые домены
2. При появлении ссылки бот получает вебхук `"event": "link_shared"` с массивом `links`
3. Извлеки данные из своей системы по URL из `links`
4. Отправь превью-данные

> Эндпоинт привязан к конкретному сообщению. Необходим специальный Unfurl-бот с указанными доменами.

**Обработать нажатие кнопки (callback)**

1. Получи вебхук с `"event": "message_button_clicked"` — в payload: `data`, `user_id`, `message_id`
2. Выполни нужное действие (запись в БД, запрос к API и т.д.)
3. Ответь пользователю
4. Опционально: обнови исходное сообщение

> Кнопка с `data` отправляет событие на вебхук. Кнопка с `url` — открывает ссылку (вебхука не будет).

**Периодический дайджест/отчёт**

1. По расписанию (cron/scheduler): собери данные из своей системы
2. Сформируй текст сообщения с нужными метриками или сводкой
3. Отправь сообщение в канал

> Нет встроенного планировщика — используй cron, celery, sidekiq и т.п. на своей стороне.

**Мониторинг и алерты**

1. Внешняя система обнаруживает событие (ошибка, деплой, порог метрики)
2. Делает POST запрос к боту или напрямую вызывает Pachca API
3. Отправь алерт с кнопками в канал
4. При нажатии кнопки — обработай callback и обнови статус алерта

**Обработка событий через историю (polling)**

1. В настройках бота включи «Сохранять историю событий». Webhook URL указывать не обязательно.
2. Получи накопленные события
3. Обработай каждое событие (тот же формат, что и в real-time вебхуке)
4. Удали обработанное событие

> Polling — альтернатива real-time вебхуку, если у бота нет публичного URL.

### pachca-forms

**Показать интерактивную форму пользователю**

1. Подготовь объект формы: `view` с `title`, `blocks`, опционально `callback_id` и `private_metadata`
2. Отправь сообщение с кнопкой
3. При нажатии кнопки — получи вебхук-событие с `trigger_id`
4. Немедленно открой форму
5. При отправке формы получи вебхук — обработай по сценарию «Обработать отправку формы»

> Формы работают только от бота.

**Обработать отправку формы (view_submission)**

1. Получи вебхук с `"type": "view"`, `"event": "submit"` — содержит `callback_id`, `user_id`, `private_metadata` и `data`
2. Извлеки значения из `data`
3. Если есть `file_input` — скачай файлы по `data.field_name[].url` немедленно
4. Если данные валидны → ответь HTTP 200 (пустое тело) — форма закроется
5. Если есть ошибки → ответь HTTP 400 с `{"errors": {"field_name": "текст ошибки"}}`

> Ответ должен быть дан в течение 3 секунд. `private_metadata` — контекст, до 3000 символов.

**Опрос сотрудников через форму**

1. Отправь сообщение с кнопкой «Пройти опрос»
2. При нажатии кнопки получи вебхук с `trigger_id`
3. Открой форму с полями опроса
4. При submit-вебхуке обработай ответы
5. Сохрани в базу или отправь итоговым сообщением в канал
6. Ответь HTTP 200 — форма закроется

> Каждый пользователь должен нажать кнопку сам — у каждого свой `trigger_id`.

**Форма заявки/запроса**

1. Размести сообщение с кнопкой «Создать заявку»
2. При нажатии открой форму с полями: тема, описание, приоритет
3. При submit: создай задачу или отправь уведомление ответственному
4. Отправь подтверждение автору
5. Ответь HTTP 200 — форма закроется

### pachca-tasks

**Создать напоминание**

1. Создай задачу

> Тип `custom_properties[].value` всегда строка. Дополнительные поля: GET /custom_properties?entity_type=Task.

**Получить список предстоящих задач**

1. Получи все задачи, фильтруй по `status` на клиенте

**Получить задачу по ID**

1. Получи информацию о задаче

**Отметить задачу выполненной**

1. Обнови статус задачи

**Обновить задачу (перенести срок, сменить ответственных)**

1. Обнови нужные поля задачи

**Удалить задачу**

1. Удали задачу

> Удаление необратимо. Чтобы просто закрыть — используй PUT с `"status": "done"`.

**Создать серию напоминаний**

1. Подготовь список дат (ежедневно, еженедельно и т.д.)
2. Для каждой даты: создай задачу

### pachca-search

**Найти сообщение по тексту**

1. Полнотекстовый поиск по сообщениям

> Поиск по всем доступным чатам. `root_chat_id` в ответе — корневой чат для тредов.

**Найти чат по названию**

1. Полнотекстовый поиск по чатам

**Найти сотрудника по имени**

1. Полнотекстовый поиск по сотрудникам

> Поиск по имени, email, должности и другим полям. Поддерживает сортировку по релевантности.

### pachca-security

**Получить журнал аудита событий**

1. Получи журнал аудита

> Доступно только владельцу пространства.

**Мониторинг подозрительных входов**

1. Получи события неудачных 2FA за период
2. Если найдены аномалии — отправь уведомление администратору

**Экспорт логов за период**

1. Получи все события за период с пагинацией
2. Собери все события в массив → сохрани в файл или отправь во внешнюю систему
