# Публикация пакетов и версии

Как выходят в реестры публикуемые пакеты репозитория и какие проверки это
гарантируют. Цель — никаких «молчаливых» релизов: всё, что попадает в `main`
и triggers публикацию, обязано нести запись в changelog, которую увидят
пользователи, и корректную версию.

Публикуемые пакеты:

| Пакет                                        | Реестр             | Версионирование                   | Свой changelog                         |
| -------------------------------------------- | ------------------ | --------------------------------- | -------------------------------------- |
| `@pachca/cli`                                | npm                | CalVer `ГГГГ.М.патч` (`2026.5.6`) | `packages/cli/src/data/changelog.json` |
| `n8n-nodes-pachca`                           | npm                | semver                            | `integrations/n8n/CHANGELOG.md`        |
| `@pachca/generator`                          | npm                | semver                            | — (только портал)                      |
| `@pachca/sdk` (TS/Python/Go/Kotlin/C#/Swift) | npm/PyPI/NuGet/git | semver, из `typespec.tsp`         | — (только портал)                      |

«Портал» — `apps/docs/data/releases.json`: единый список релизов всех
продуктов, он же лента `/updates` на сайте. Формат записей — в
`docs/updates-format.md`.

## Единый источник версии

Версия пакета объявляется **вручную** в `releases.json` (запись с нужным
`product` и `version`) и, для CLI/n8n, продублирована в собственном changelog
пакета. Никакого авто-инкремента из npm — раньше он молча переиздавал
идентичный код под новым номером (так появились лишние `cli 2026.5.5/5.6` и
`generator 1.1.3/1.1.4`).

Версия в `releases.json` и в собственном changelog **обязаны совпадать** —
это проверяется на сборке (см. ниже).

Исключение — SDK: его версия берётся из `packages/spec/typespec.tsp`, а все
6 языков публикуются одной версией вместе. В `releases.json` SDK всё равно
присутствует (для ленты и для согласованности).

## Две проверки

### 1. Сборка PR — `scripts/check-changelog-sync.mjs`

Запускается в workflow `.github/workflows/check.yml` на каждый PR (и локально
`bun run check:changelog`). Сравнивает ветку с `origin/main` и **роняет
сборку**, если для любого пакета:

- код пакета изменился, но нет записи в changelog (releases.json + own
  changelog) — запрещает молчаливый релиз;
- запись в changelog добавлена, но код пакета не менялся;
- версия в `releases.json` не совпадает с версией в собственном changelog.

«Код пакета» — любой файл в директории пакета, кроме самих changelog-файлов.

### 2. Публикация — `scripts/check-release.mjs`

Запускается в publish-workflow (`sdk.yml` для CLI и SDK, `n8n.yml`,
`generator.yml`) на `push` в `main`. Пакет публикуется, только если **все**
условия выполнены для версии из `releases.json`:

1. **Формат** — CLI ровно `ГГГГ.М.патч`, остальные `X.Y.Z`.
2. **Точный следующий шаг** относительно максимальной версии в npm (не просто
   «больше»): semver `патч+1` / `минор+1.0` / `мажор+1.0.0`; CalVer `патч+1` в
   том же месяце или `.0` в новом месяце. Перепрыг (`1.1.4 → 1.1.6`) запрещён.
3. **Нет в npm** — идемпотентность: повторный запуск не переиздаёт.
4. **Код пакета менялся** в этом push.
5. **Версия есть в собственном changelog** (CLI/n8n).

Если версия не проходит шаг (2), gate печатает допустимые варианты, например:
`generator: version "1.1.6" is not a valid next step after npm 1.1.4.
Allowed: 1.1.5 (patch), 1.2.0 (minor), 2.0.0 (major)`.

Шаг считается **от того, что реально в npm**, а не от `releases.json`. Если
баг прошлого занял номера (generator в npm уже `1.1.4` при портале `1.1.2`),
следующий релиз = `1.1.5` — занятые номера переиздать нельзя.

## Как выпустить релиз

1. Внеси изменения в код пакета (или в `typespec.tsp` для SDK).
2. Узнай максимальную версию в npm: `npm view <pkg> version`
   (CLI `@pachca/cli`, n8n `n8n-nodes-pachca`, generator `@pachca/generator`,
   SDK `@pachca/sdk`).
3. Вычисли следующую версию по правилу шага (см. проверку 2). Для SDK — подними
   версию в `packages/spec/typespec.tsp`.
4. Добавь запись в `apps/docs/data/releases.json` (новая запись сверху, формат
   — `docs/updates-format.md`).
5. Для CLI добавь ту же версию в `packages/cli/src/data/changelog.json`; для
   n8n — заголовок `## X.Y.Z (дата)` в `integrations/n8n/CHANGELOG.md`.
   Версии обязаны совпадать с `releases.json`.
6. Открой PR. `check.yml` проверит синхронность код↔changelog.
7. После merge в `main` publish-workflow опубликует ровно объявленную версию
   (один раз; повторные push не переиздают).

Локально перед PR: `bun run check:changelog`.

## Чего НЕ делать

- Не публиковать руками `npm publish` — только через workflow.
- Не вписывать произвольную версию: gate требует точный следующий шаг от
  npm-max. Кривую/перепрыгнутую версию он отклонит.
- Не менять код пакета без записи в changelog (и наоборот) — сборка упадёт.
- Не редактировать `packages/cli/CHANGELOG.md` руками — он генерируется из
  `changelog.json` при сборке CLI.
- `@pachca/generator` `1.1.3/1.1.4` и `@pachca/cli` `2026.5.5` — артефакты
  старого авто-инкремента (до этих проверок). Не пытаться «переиздать» их.
